Estudo — Playwright E2E Testing

Cada arquivo explicado, cada decisão justificada, com equivalentes em Node.js pra entrevista.

O que é esse projeto

Suite de 33 testes E2E automatizados pro site The Internet (the-internet.herokuapp.com), um site de prática pra automação criado pelo autor do Selenium.

Testa 9 funcionalidades diferentes: login, checkboxes, dropdown, drag-and-drop, carregamento dinâmico, upload de arquivo, hovers, teclas e alertas JavaScript.

Arquivo 1/3

pytest.ini — Configuração central

[pytest]
base_url = https://the-internet.herokuapp.com
testpaths = tests
base_url: A URL base do site. Quando faço page.goto("/login"), o pytest-playwright concatena automaticamente com essa URL. Evita repetir a URL completa em todo teste.

testpaths: Diz ao pytest pra procurar testes só na pasta tests/. Sem isso, ele varreria todo o projeto.
Em Node.js: O equivalente é o playwright.config.ts: 
export default defineConfig({
  use: {
    baseURL: 'https://the-internet.herokuapp.com',
  },
  testDir: './tests',
});
Se perguntarem: "Configuração minimalista de propósito. O pytest-playwright gerencia browser, contexto e page automaticamente. Timeout, retries e tracing passo via linha de comando."

requirements.txt — Dependências

Só 3 pacotes. O pytest-playwright é a cola entre Playwright e pytest — fornece a fixture page, gerencia o browser, e configura a base_url.

Arquivo 3/3

conftest.py — Fixtures globais

import pytest

from pages.login_page import LoginPage
# ... (importa todos os 9 page objects)

@pytest.fixture()
def login_page(page):
    return LoginPage(page)

@pytest.fixture()
def checkboxes_page(page):
    return CheckboxesPage(page)

# ... (uma fixture pra cada page object)
O que é conftest.py: Arquivo especial do pytest. Tudo que estiver aqui fica disponível automaticamente pra todos os testes, sem precisar importar.

O que é uma fixture: Uma função que prepara algo pro teste. Aqui, cada fixture recebe page (que o pytest-playwright injeta — é uma instância limpa do browser) e retorna o page object pronto.

O fluxo completo:
1. O teste pede login_page como parâmetro.
2. O pytest procura no conftest.py e acha a fixture.
3. A fixture pede page — o pytest-playwright injeta um browser limpo.
4. A fixture cria LoginPage(page) e devolve pro teste.
5. O teste usa o page object sem saber nada de browser ou seletores.

Por que cada teste recebe um page novo: Isolamento. Cada teste começa do zero, sem estado residual do teste anterior. Se um teste falha, não afeta os outros.
Em Node.js: O Playwright Test tem fixtures embutidas. Tu usaria test.extend(): 
import { test as base } from '@playwright/test';
import { LoginPage } from './pages/login-page';

export const test = base.extend({
  loginPage: async ({ page }, use) => {
    await use(new LoginPage(page));
  },
});
Se perguntarem "o que é conftest.py": "É o arquivo onde defino fixtures compartilhadas. Cada page object vira uma fixture que o pytest injeta automaticamente nos testes pelo nome do parâmetro. A fixture page vem do pytest-playwright e me dá um browser limpo pra cada teste."

Conceito central

Page Object Model (POM)

Cada página do site tem uma classe Python com:
- Seletores como constantes de classe (ex: USERNAME_INPUT = "#username")
- Ações como métodos (ex: login(), visit())
- Recebe page no construtor e guarda em self.page

O teste nunca acessa seletores diretamente. Sempre passa pelo page object.

Sem POM:

page.fill("#username", "tomsmith")
page.fill("#password", "SuperSecretPassword!")
page.click('button[type="submit"]')

Com POM:

login_page.login("tomsmith", "SuperSecretPassword!")
Vantagens:
1. Se o seletor #username mudar pra #user-input, altera em um arquivo só.
2. Testes ficam legíveis — login_page.login() é mais claro que 3 linhas de page.fill().
3. Reutilização — o método login() é usado em vários testes.
Se perguntarem "por que POM": "Centralizo seletores e ações num lugar só. Se a UI muda, atualizo o page object e todos os testes continuam funcionando. Facilita manutenção e deixa os testes legíveis."

Page Object 1/9

login_page.py — O mais completo

class LoginPage:
    # Seletores como constantes de classe
    USERNAME_INPUT = "#username"
    PASSWORD_INPUT = "#password"
    SUBMIT_BUTTON = 'button[type="submit"]'
    FLASH_MESSAGE = "#flash"
    LOGOUT_BUTTON = 'a[href="/logout"]'

    def __init__(self, page):
        self.page = page          # Guarda a instância do browser

    def visit(self):
        self.page.goto("/login")  # Usa base_url do pytest.ini

    def fill_username(self, username):
        self.page.fill(self.USERNAME_INPUT, username)

    def fill_password(self, password):
        self.page.fill(self.PASSWORD_INPUT, password)

    def submit(self):
        self.page.click(self.SUBMIT_BUTTON)

    def login(self, username, password):
        # Método combo — faz tudo de uma vez
        self.fill_username(username)
        self.fill_password(password)
        self.submit()

    def get_flash_message(self):
        # Retorna um LOCATOR, não texto
        return self.page.locator(self.FLASH_MESSAGE)

    def logout(self):
        self.page.click(self.LOGOUT_BUTTON)
Decisões importantes:

1. Seletores como constantes de classe (não no construtor): São compartilhados entre todas as instâncias. Não mudam. Fácil de encontrar e atualizar.

2. self.page: No Playwright Python, page não é global (diferente do cy no Cypress). Precisa ser passado no construtor e armazenado.

3. Métodos granulares + combo: fill_username(), fill_password() e submit() existem separados pra testes que precisam testar campos individuais (ex: campos vazios). O login() é um atalho pra testes que precisam do fluxo completo.

4. get_flash_message() retorna um locator: Não retorna texto direto. Retorna o locator (objeto do Playwright que representa o elemento). Isso permite usar expect() com auto-retry depois. Se retornasse texto, não teria retry.

Page Object 2/9

checkboxes_page.py — Indexação de elementos

class CheckboxesPage:
    CHECKBOXES = '#checkboxes input[type="checkbox"]'

    def get_checkbox(self, index):
        return self.page.locator(self.CHECKBOXES).nth(index)

    def toggle(self, index):
        self.get_checkbox(index).click()
Conceito novo: .nth(index)
O seletor #checkboxes input[type="checkbox"] retorna TODOS os checkboxes. O .nth(0) pega o primeiro, .nth(1) o segundo. É como acessar um array por índice.

Toggle é um click: Marcar e desmarcar um checkbox é a mesma ação — clicar. O estado muda sozinho.

Page Object 3/9

dropdown_page.py — Select nativo

def select_option(self, value):
    self.page.select_option(self.DROPDOWN, value)
page.select_option() é o método nativo do Playwright pra elementos <select>. Pode selecionar por value ("1"), por label (label="Option 1"), ou por index.

Page Object 4/9

drag_and_drop_page.py — O mais complexo

def drag_a_to_b(self):
    self.page.evaluate("""() => {
        const source = document.querySelector('#column-a');
        const target = document.querySelector('#column-b');
        const dataTransfer = new DataTransfer();
        source.dispatchEvent(new DragEvent('dragstart',
            { dataTransfer, bubbles: true }));
        target.dispatchEvent(new DragEvent('dragover',
            { dataTransfer, bubbles: true }));
        target.dispatchEvent(new DragEvent('drop',
            { dataTransfer, bubbles: true }));
        source.dispatchEvent(new DragEvent('dragend',
            { dataTransfer, bubbles: true }));
    }""")
Por que não usar drag_to()?
O site usa HTML5 Drag and Drop API, que depende de eventos DataTransfer específicos. O page.locator().drag_to() do Playwright simula eventos de mouse, mas esses não disparam os eventos HTML5 DnD. É um problema conhecido.

O workaround: page.evaluate() executa JavaScript direto no browser. A gente dispara manualmente os 4 eventos do ciclo de drag: dragstart → dragover → drop → dragend.

page.evaluate() é o "escape hatch" do Playwright — quando o framework não consegue fazer algo nativamente, tu executa JS direto no contexto da página. Deve ser último recurso.
Se perguntarem sobre um desafio técnico: "O drag-and-drop do site usava HTML5 DnD API, que não responde aos eventos do Playwright. Tive que usar page.evaluate() pra disparar os eventos nativos via JavaScript. Documentei como bug e como workaround."

Page Object 5/9

dynamic_loading_page.py — Auto-waiting

def visit_example(self, example_number):
    self.page.goto(f"/dynamic_loading/{example_number}")

def get_loaded_text(self):
    return self.page.locator(self.LOADED_TEXT)
Dois exemplos no site:
- Exemplo 1: Elemento já existe no HTML mas está oculto (CSS display: none). Após loading, fica visível.
- Exemplo 2: Elemento NÃO existe no HTML. É criado dinamicamente após loading.

O Playwright lida com os dois automaticamente. Quando faço expect(locator).to_be_visible(), ele espera até o elemento aparecer. Sem time.sleep(), sem wait explícito. Isso é o auto-waiting.
Se perguntarem "como lida com elementos que demoram a aparecer": "O Playwright tem auto-waiting nativo. Ele espera o elemento ficar visível, habilitado e estável antes de interagir. Não preciso de sleep ou wait explícito. Se preciso de um timeout maior, passo como parâmetro: to_be_visible(timeout=10000)."

Page Object 6/9

file_upload_page.py — Upload sem diálogo

def upload_file(self, file_path):
    self.page.set_input_files(self.FILE_INPUT, file_path)
    self.page.click(self.SUBMIT_BUTTON)
page.set_input_files() define o arquivo no input type="file" diretamente, sem abrir o diálogo do sistema operacional. Essencial pra automação — diálogos nativos do SO são difíceis de controlar.

Page Object 7/9

hovers_page.py — Locator encadeado

def get_figure_caption(self, index):
    return self.page.locator(self.FIGURES).nth(index).locator(self.FIGURE_CAPTION)
Locator encadeado: Primeiro pega o container (.figure) pelo índice, depois busca dentro dele (.figcaption h5). Evita pegar o caption do avatar errado.

Page Object 8/9

key_presses_page.py — Simulação de teclado

def press_key(self, key):
    self.page.press(self.INPUT_TARGET, key)
page.press() simula pressionar uma tecla. Aceita nomes como 'Enter', 'Tab', 'Escape', 'Space', ou letras como 'a'. O Enter foi removido dos testes porque submitava o formulário em vez de registrar como keypress.

Page Object 9/9

javascript_alerts_page.py — Seletores por texto

ALERT_BUTTON = 'button:text("Click for JS Alert")'
CONFIRM_BUTTON = 'button:text("Click for JS Confirm")'
PROMPT_BUTTON = 'button:text("Click for JS Prompt")'
Text selector: button:text("...") busca o botão pelo texto visível. Mais resiliente que seletor por posição (ex: button:nth-child(1)). Se a ordem dos botões mudar, o teste não quebra.

Os handlers de diálogo ficam nos testes, não aqui. Porque o handler precisa ser registrado ANTES do clique que dispara o alert — então faz mais sentido ficar no contexto do teste.

Estrutura base

Como um teste funciona

from playwright.sync_api import expect

class TestLogin:
    def test_should_login_with_valid_credentials(self, login_page, page):
        login_page.visit()
        login_page.login("tomsmith", "SuperSecretPassword!")

        expect(page).to_have_url(re.compile(r"/secure"))
        expect(login_page.get_flash_message()).to_contain_text(
            "You logged into a secure area!"
        )
Classes agrupam testes (equivalente ao describe do Cypress/JS).
Métodos test_* são os testes (equivalente ao it).
Parâmetros são fixtures injetadas: login_page vem do conftest.py, page do pytest-playwright.
expect() vem do playwright.sync_api — tem auto-retry com timeout. Diferente de assert puro do Python que falha instantaneamente.
to_contain_text em vez de to_have_text porque a mensagem vem com um "x" de fechar no final.

test_dynamic_loading.py — Auto-waiting na prática

def test_should_show_hidden_element(self, dynamic_loading_page):
    dynamic_loading_page.visit_example(1)
    dynamic_loading_page.click_start()

    # Timeout explícito porque o loading demora
    expect(dynamic_loading_page.get_loaded_text()).to_be_visible(timeout=10000)
    expect(dynamic_loading_page.get_loaded_text()).to_have_text("Hello World!")
Por que timeout=10000? O default do Playwright é 5 segundos. O loading do site demora mais, então aumento pra 10s. É o único lugar onde preciso de timeout explícito — em todo o resto, o auto-waiting padrão dá conta.

test_hovers.py — Loop em testes

def test_should_have_correct_profile_links(self, hovers_page):
    hovers_page.visit()
    for i in range(3):
        hovers_page.hover_over_figure(i)
        expect(hovers_page.get_profile_link(i)).to_have_attribute(

Loop valida que cada avatar linka pro perfil correto: user1 → /users/1, user2 → /users/2, etc. O f"/users/{i + 1}" é uma f-string do Python (template literal do JS).

test_javascript_alerts.py — Diálogos (o mais importante)

def test_should_accept_js_alert(self, javascript_alerts_page, page):
    javascript_alerts_page.visit()

    # PRIMEIRO registra o handler
    page.on("dialog", lambda dialog: dialog.accept())

    # DEPOIS clica no botão que abre o alert
    javascript_alerts_page.trigger_alert()

    expect(javascript_alerts_page.get_result()).to_have_text(
        "You successfully clicked an alert"
    )
REGRA CRÍTICA: O handler de diálogo precisa ser registrado ANTES do clique que dispara o alert. Se inverter a ordem, o alert aparece e o Playwright não sabe o que fazer.

Três tipos de diálogo:
- Alert: Só "OK". Usa dialog.accept()
- Confirm: "OK" ou "Cancel". Usa dialog.accept() ou dialog.dismiss()
- Prompt: Campo de texto + "OK"/"Cancel". Usa dialog.accept('texto') pra enviar texto

lambda: Função anônima do Python. lambda dialog: dialog.accept() é o mesmo que definir uma função que recebe dialog e chama .accept().
Em Node.js: Mesmo conceito, sintaxe diferente: 
page.on('dialog', dialog => dialog.accept());
// ou com async:
page.on('dialog', async dialog => {
  await dialog.accept('texto');
});
Se perguntarem como lida com alerts: "Registro o handler ANTES do clique que dispara o alert. No Playwright, uso page.on('dialog') com uma callback que aceita ou rejeita o diálogo. A ordem é importante — se registrar depois, o alert já apareceu e o teste trava."

GitHub Actions — Pipeline

name: Playwright Tests

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4           # 1. Baixa o código

      - uses: actions/setup-python@v5       # 2. Instala Python 3.12
        with:
          python-version: "3.12"
          cache: pip

      - run: pip install -r requirements.txt # 3. Instala dependências

      - run: python -m playwright install    # 4. Instala Chromium
            --with-deps chromium

      - run: python -m pytest -v             # 5. Roda os testes
            --tracing=retain-on-failure

      - uses: actions/upload-artifact@v4     # 6. Salva resultados
        if: always()
        with:
          name: test-results
          path: test-results/
          retention-days: 30
Quando roda: A cada push e pull request na branch main.

O que faz, passo a passo:
1. Checkout — baixa o código do repositório.
2. Setup Python — instala Python 3.12 com cache do pip (mais rápido nas próximas vezes).
3. Install — instala playwright, pytest e pytest-playwright.
4. Playwright install — baixa o Chromium e dependências gráficas do Linux (o ubuntu-latest não tem browser).
5. Pytest — roda todos os testes. -v = verbose. --tracing=retain-on-failure = grava trace quando falha.
6. Upload artifact — salva os resultados (traces) como artifact no GitHub. if: always() = salva mesmo se o teste falhar. Ficam disponíveis por 30 dias.

Traces: Quando um teste falha, o Playwright grava um arquivo .zip com screenshots, rede, DOM e timeline. Tu pode abrir no trace.playwright.dev pra debugar.
Em Node.js: A pipeline seria quase igual, trocando Python por Node: 
- uses: actions/setup-node@v4
- run: npm ci
- run: npx playwright install --with-deps
- run: npx playwright test
Se perguntarem sobre CI/CD: "Configurei uma pipeline com GitHub Actions que roda os testes automaticamente a cada push. Instala dependências, baixa o browser, roda todos os testes e salva traces das falhas como artifacts. A ideia é garantir que nenhuma mudança quebre o que já funciona antes de chegar em produção."

import { test, expect } from '@playwright/test';
import { LoginPage } from './pages/login-page';

test('should login with valid credentials', async ({ page }) => {
    const loginPage = new LoginPage(page);
    await loginPage.visit();

Diferenças:
1. async/await em tudo no Node.js.
2. test() em vez de class + def.
3. toHaveURL em vez de to_have_url.
4. Regex direto (/\/secure/) em vez de re.compile().
5. Page object criado dentro do teste (ou via fixture com test.extend).

O que é igual: Seletores, lógica, expect, locators, auto-waiting. Tudo.

Exemplo lado a lado — Diálogos

Python

page.on("dialog", lambda dialog: dialog.accept("Hello"))
javascript_alerts_page.trigger_prompt()

Node.js

page.on('dialog', dialog => dialog.accept('Hello'));
await javascriptAlertsPage.triggerPrompt();
Praticamente idêntico. lambda vira arrow function, e adiciona await.

Exemplo lado a lado — Page Object

Python

class LoginPage:
    USERNAME_INPUT = "#username"

    def __init__(self, page):
        self.page = page

    def fill_username(self, username):
        self.page.fill(self.USERNAME_INPUT, username)

Node.js / TypeScript

export class LoginPage {
    private page: Page;
    private USERNAME_INPUT = '#username';

    constructor(page: Page) {
        this.page = page;
    }

    async fillUsername(username: string) {
        await this.page.fill(this.USERNAME_INPUT, username);
    }
}
TypeScript adiciona: tipos (: Page, : string), visibilidade (private), e export. A lógica é idêntica.

Perguntas técnicas que o gestor pode fazer

Lembra: A entrevista técnica é com o gestor. Ele provavelmente vai pedir pra tu explicar teu código, escrever algo, ou resolver um cenário.
"Me explica a arquitetura do teu projeto"
Page Object Model: cada página tem uma classe com seletores e ações. Os testes usam os page objects via fixtures do pytest, sem tocar em seletores. O conftest.py injeta os page objects automaticamente. CI/CD com GitHub Actions roda tudo a cada push.
"Por que Page Object Model e não outra coisa?"
Centraliza seletores num lugar só — se a UI muda, atualizo uma classe e todos os testes continuam. Deixa os testes legíveis (leio loginPage.login() em vez de 3 linhas de fill/click). É o padrão mais usado na indústria pra testes E2E.
"O que faria diferente se refizesse o projeto?"
Usaria TypeScript com Node.js em vez de Python, já que é a stack mais comum no mercado. Adicionaria testes em paralelo pra rodar mais rápido. E usaria fixtures mais sofisticadas pra dados de teste (factories em vez de dados hardcoded).
"Como tu debugaria um teste que tá falhando?"
1. Leio a mensagem de erro — o Playwright diz exatamente qual expect falhou e por quê. 2. Rodo com --headed pra ver o browser executando. 3. Se precisar mais detalhe, rodo com --tracing=on e abro o trace no trace.playwright.dev — mostra screenshots, rede, DOM e timeline de cada ação. 4. Se for um problema de timing, aumento o timeout ou investigo se é um flaky test.
"Escreve um teste pra [cenário X]"
Se pedirem pra escrever ao vivo, segue o padrão:
1. Cria o page object (classe com seletores e ações).
2. Cria a fixture no conftest (ou inline se for rápido).
3. Escreve o teste: visit → ação → expect.
Pensa em voz alta enquanto escreve — mostra raciocínio.
"Como tu lida com elementos que não estão na tela ainda?"
O Playwright tem auto-waiting nativo. Quando chamo uma ação (click, fill) ou uma assertion (expect), ele espera automaticamente até o elemento estar visível, habilitado e estável. Se preciso de mais tempo, passo timeout como parâmetro. Nunca uso sleep.
"O que é um flaky test e como resolve?"
Teste que às vezes passa e às vezes falha sem mudança no código. Geralmente é: timing (elemento não carregou), dados compartilhados entre testes, ou race condition. Resolvo garantindo isolamento entre testes (cada um tem seu page limpo), usando auto-waiting em vez de timeouts fixos, e evitando dependência de ordem de execução.
"Como testaria uma funcionalidade com muitos cenários?"
Usaria partição de equivalência pra agrupar cenários similares e testar um de cada grupo. Depois análise de valor limite pros extremos. Automatizaria o happy path e os cenários críticos, deixaria edge cases exóticos pra teste exploratório manual.
"Tu sabe o que é o playwright.config.ts?"
É a configuração central do Playwright em Node.js — equivalente ao meu pytest.ini + conftest.py. Define baseURL, browsers, timeout, retries, reporter, testDir, e pode configurar múltiplos projects (ex: rodar em Chrome e Firefox).
